    • Failure

    Failure can happen during computation for a variety of reasons. Division by zero, for example, can cause failure. A more benign reason might be that a particular branch of a computation has not lead to a solution, and so this branch should be abandoned.

    Failure causes the current thread to stop. The termination of this thread can trigger other threads to start, or can cause the entire program to be stopped.

    Exceptions

    Whenever a failure happens, the failure has a value associated with it called an exception. The exception gives some indication of why the failure occurred. For example, if you try to take the head of an empty list, the head function will cause a failure, with exception emptyListX.

    An exception is a member of species ExceptionSpecies. Standard exceptions include emptyListX (which occurs when you try to take the head or tail of an empty list) and testX (which occurs when a test fails, in cases where such failure is expected to occur in the normal course of computation). Some exceptions are created by constructors with parameters, such as domainX("..."). For example, when you divide by 0, computation fails with exception domainX("divide by 0").

    When a failure occurs, the exception is either trapped (causing computation to stop immediately) or is not trapped. See trapping, below.

    The following is a list of the standard exceptions, with indications of their meanings. More detail can be found in lang/except.ast

    Standard exceptions

    badInputX
    Input was of the wrong form. Normally not trapped.

    bindUnknownX
    Failure to bind an unknown. Normally not trapped.

    cannotSeekX
    Attempt to seek on an unseekable file. Normally not trapped.

    closedFileX
    Attempt to write to a closed file. Normally not trapped.

    closedPipeX
    Attempt to write to a closed pipe. Normally not trapped.

    conversionX
    Impossible conversion. Normally not trapped.

    domainX("explanation")
    Value is not in the domain of a function. Normally trapped.

    emptyBoxX
    Attempt to take content of an empty box. Normally trapped.

    emptyListX
    Attempt to take head or tail of an empty list. Normally trapped.

    endThreadX
    A thread terminated itself. Normally not trapped.

    This exception is not caught by most try expressions. See catchingTermination.

    ensureX("explanation")
    A function failed to ensure a condition that it should ensure is true. Normally trapped.

    infiniteLoopX
    A lazy expression tried to use its own value. Normally trapped.

    interruptX
    The program was interrupted by the user. Normally trapped.

    limitX
    A built-in limit has been exceeded. Normally trapped.

    listExhaustedX
    An attempt has been made to extract a value from a string that is empty, or contains only white space. Normally trapped.

    memoryX
    The system has run out of memory. Normally trapped.

    noCaseX
    There was no applicable case in a choose or loop expression. Normally trapped.

    noDefX("reason")
    No definition was found for an identifier. Normally trapped.

    noFileX("file")
    File could not be opened. Normally not traped.

    requireX("explanation")
    A precondition of a function was violated. Normally trapped.

    sizeX
    Something was the wrong size. Normally trapped.

    speciesX
    There was a run-time species mismatch. Normally not trapped.

    subscriptX
    Subscript out of bounds. Normally trapped.

    systemX
    A system function failed. Normally trapped.

    terminateX
    This thread was terminated by another thread. Normally not trapped.

    This exception is not caught by most try expressions. See catchingTermination.

    testX
    A test failed. Normally not trapped.

    tooManyFilesX
    There are too many open files. Normally trapped.

    Testing and breaking apart exceptions

    You can test an exception to see whether it is of a given kind. Add a question mark to the end of the exception constructor. For example, emptyListX?(e) is true just when e is emptyListX, and domainX?(e) is true just when e has kind domainX.

    You can get the string associated with a domainX, ensureX, etc. value either by using pattern matching or by using the functional inverse of the constructor, which begins with "un" For example, undomainX(domainX(s)) = s. Other functions are unensureX, unnoDefX, unnoFileX and unrequireX.

    Causing failure

    The library functions can fail under some circumstances. For example, if you divide by zero, the division function causes the thread to fail. You can force failure any time as follows.

    fail e
    Library function fail takes an exception as an argument, and causes a failure with that exception. Expression fail(e) can have any type; it never returns a value though, since it fails.

    Example: fail(emptyListX)

    Fail e %Fail
    Library function Fail is the same as fail, but its result type is ().

    Example: Fail emptyListX .

    {a else b}
    {a else b} has no effect if a evaluates to true, but fails with exception b if a evaluates to false. Expression b must evaluate to an exception.

    Example: {a == b else emptyListX}

    {a}
    {a} abbreviates {a else testX}.

    Catching failure

    A program can recover from a failure by catching it. The following mechanisms are provided for catching failure.

    Try a then b else c %Try
    This expression catches a failure in expression a. It has the same value as b if evaluation of a succeeds, and the same value as c if evaluation of a fails. Expression a must have species ().

    A failure is caught by the most recently entered construct (such as a try) that can catch an failure. For example, 

       Try
     Try
       Fail testX.
     then
       Writeln["This won't be printed"].
     else
       Writeln["This will be printed"].
     %Try
   then
     Writeln["This will be printed too"].
   else
     Writeln["This won't be printed"].
   %Try
    prints 
        This will be printed
    This will be printed too
    since the inner try catches the failure. The inner try does not need to be statically (in the program structure) inside the outer try. It must be dynamically inside it, merely entered more recently during computation.

    a _butIfFail_ b
    If evaluation of expression a succeeds, then a _butIfFail_ b has the same value as a. If evaluation of a fails, then a _butIfFail_ b has the same value as b.

    a_butIfFail_b is translated to 

      Try catchingEachThread 
      Let x = a. 
    then x 
    else b 
  %Try
    where x is a new identifier.

    exception
    Within the else-part of a try-expression, you can use expression exception. Its value is the exception that was caused in the condition of the try-expression.

    For example, the following recovers from a failure of e having exception emptyListX, but just propagates any other exception outwards, continuing the failure. 

      Try
    e
  then
    ...
  else
    If emptyListX?(exception) then
      Writeln["Ok, let's just keep going"].
    else
      Fail exception
    %If
  %Try
    Note: A choose-expression is translated into a collection of if-expressions and try-expresions. You can use exception in a choose-expression in parts that follow a try-expression in the translation. A try-expression is used any time the choose-expression is checking for failure, including when pattern matching is used. For example 
         Choose first matching x
       case [] => ...
       do Writeln[$exception].
       ...
     %Choose
    writes "testX"' when the first case fails, and then continues testing more cases.

    Miscellaneous operations

    succeedsf
    : (`a -> `b) -> `a -> Boolean

    Expression (succeedsf f x) is true if f(x) does not fail.

    willTrap
    : ExceptionSpecies -> Boolean

    Expression willTrap(e) is true if exception e will be trapped if it occurs in the current context.

    Handling multiple threads

    Try expression have options for controlling how they handle forking of threads. Please read about threads before reading this.

    Try catchingTermination ... %Try
    Normally, a try expression will not catch exceptions terminateX or endThreadX, since those exceptions are supposed to destroy a thread. Sometimes, you need to catch them, so that a thread can clean up when it is destroyed. If you add modifier catchingTermination after Try, then endThreadX and terminateX will be caught. If a try catches termination, it should propagate the same exception outwards after handling it. For example, a try expression might look like this. 
        Try catchingTermination
      ...
    else
      clean up
      Fail exception.
    %Try
    If, after catching termination, a thread continues running normally, then it can, conceivably, reach the end of a Unique expression, possibly causing more than one thread to exit an expression that is only supposed to have one thread exit.

    Try catchingEachThread ... %Try
    Consider a try-expression 
       Try
     a
   then
     b
   else
     c
   %Try
    When a thread enters this try-expression, another thread is created to handle the failure, and is immediately suspended. That thread will only be started when execution of a fails. Let's call the waiting thread W.

    Suppose that a single thread enters this try expression, but that new threads are created inside a. Call two of those threads G (for the good thread) and B (for the bad thread). Suppose further that thread G succeeds, and makes it to the end if a, but thread B fails, and does not make it to the end if a.

    There are two reasonable ways for the try-expression to run.

    1. Threads are often used for searching. A thread spawns several other threads to try different alternatives. Typically, some of those alternatives will not work out, and the corresponding threads will fail. A thread is considered to have failed to compute an expression only when none of the threads that it spawns succeeds.

      With this view, the try-expression should only start up the waiting thread W when all of the threads spawned by the thread that started a have failed. Since thread G succeeds, the try expression will destroy thread W without running it.

      This is the default rule for try-expressions.

    2. Threads can, alternatively, be thought of as separate agents, each performing a separate task. When thread B fails inside expression a, the try expression should catch that failure. This involves creating a copy of the waiting thread W and running that thread. (A copy must be made since, under this interpretation, many different threads might fail, and perform expression c.)

      This is the rule when option catchingEachThread is used.

    Options catchingEachThread and catchingTermination can be combined in either order after the word Try.

    Trapping and untrapping exceptions

    When a failure occurs, one of two things happens. If the exception that is associated with the failure is trapped, then the program aborts. If the exception is not trapped, then the thread that failed is stopped. Stopping this thread can make another thread become active. That other thread can handle the failure. For example, a try-expression sets up a thread that handles an exception. You can control which exceptions are trapped when the exceptions are created (see exception declarations), or dynamically using trap and untrap expressions.

    Untrap a => e %Untrap
    This evaluates expression e with trapping of exception a turned off. Some exceptions come in groups of the same kind. For example, domainX("one") and domainX("two") are both exceptions of kind domainX. If a is a member of such a group of exceptions, then trapping is suppressed for all exceptions in that group.

    Example: 

        Untrap emptyListX =>
      head(x)
    %Untrap

    Untrap a,b => e %Untrap
    This abbreviates Untrap a => Untrap b => e %Untrap %Untrap. You can have any number of exceptions to untrap.

    Untrap all => e %Untrap
    This evaluates expression e with trapping of all exceptions turned off.

    Trap a => e %Trap
    This is similar to untrap, but causes the exception to be trapped. The most recently entered trap or untrap expression has precedence.

    Trap a,b => e %Trap
    This abbreviates Trap a => Trap b => e %Trap %Trap. You can have any number of exceptions to trap.

    Trap all => e %Trap
    This causes all exceptions to be trapped while evaluating e.
