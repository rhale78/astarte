    • Defaults

    Occasionally, type inference encounters a variable that needs to be bound to a species, but for which no information is available to say how the binding should be done. In that case, a default is chosen.

    Some defaults are considered relatively benign. For example, defaulting a variable INTEGER`a to Natural rarely has much of an impact on a program. Some defaults, although they must be done, are more suspicious, and the programmer is warned about them. They are called dangerous defaults. For example, defaulting variable `a to () is considered a dangerous default.

    The standard genera and communities have defaults defined for them. The standard defaults are as follows. Defaults marked with an asterisk(*) are dangerous.

    Abstraction Default
    APPROX Real
    ANY ()*
    EQ ()*
    ORDER ()*
    RANKED Natural*
    ENUMERATED Boolean*
    FIELD Rational
    GROUP Integer
    REAL Natural
    RATIONAL Natural
    INTEGER Natural
    NFIELD Rational
    NRING Integer
    NUMBER Natural
    RATRING Integer
    RING Integer
    RRING Integer
    RFIELD Rational
    SEMIGROUP Natural
    SEMIRING Natural
    OPAQUE Box*
    OPAQUE_EQ (none)
    OPAQUE_ORDER (none)
    TRANSPARENT List*
    TRANSPARENT_EQ List*
    TRANSPARENT_ORDER List*

    Changing defaults

    You can change the default that is chosen using a default declaration. For example, 

        Default ANY => Natural.
    forces a variable with domain ANY to be defaulted to Natural when a default is necessary. In general, the form is Default G => D %Default, where G is an abstraction and D is the default species or family. Species D must be a primary species.

    To declare a dangerous default, use mode dangerous, as in the following. 

        Default{dangerous} ANY => Natural.

    Secondary defaults

    A variable such as ``a, which must be a secondary species, cannot be bound according to the defaults above. Instead, a variable that has domain G is defaulted to secondary species G. For example, variable ``a defaults to ANY.

    Normally, a variable such as `a that can be either primary or secondary chooses a primary default. You can force the default be be secondary, though, by choosing a name that starts with an upper case letter. For example, variable `A will be defaulted to ANY, when a default is needed.

    Secondary defaults have precedence over primary defaults when variables are made the same. For example, if variables `A and `b must be made equal to one another during type inference, the resulting variable prefers a secondary default.

    Constraints and run-time defaults

    When there are no constraints, all defaults are selected at compile time. The presence of constraints such as `a >= `b, forces some defaults to be chosen at run time. Normally, this should be transparent to you. If you request to see the results of type inference (via an advise declaration with option show), you will see unbound variables that will be defaulted at run time, when more type information is obtained.
