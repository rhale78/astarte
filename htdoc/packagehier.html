    • The package hierarchy

    The compiler maintains a relationship among packages called the subpackage relation. You declare a package to be a subpackage another package in the heading by including an "isKindOf" phrase. For example, 
      Package Fox isKindOf Mammal
    ...
  %Package
    indicates that package Fox is a subpackage of package Mammmal. The subpackage relation is transitive. That is, if Fox is a subpackage of Mammal and Mammal is a subpackage of Animal, then Fox is automatically a subpackage of Animal.

    A subpackage in not written in the same file as another package. The subpackage relationship goes across files.

    You can use several isKindOf phrases. For example, 

      Package Fox isKindOf Mammal isKindOf Carnivore
    ...
  %Package
    indicates that Fox is both a subpackage of Mammal and a subpackage of Carnivore. Alternatively, you can write 
      Package Fox isKindOf Mammal, Carnivore
    ...
  %Package

    Visibility and protected mode

    When a package makes a definition or expectation, it can control which other packages can see that definition or expectation. Mode protected{A,B,C} indicates that a definition or expectation is visible only to packages A, B and C, or to any of their subpackages. You can have any positive number of package names in the braces. The current package name is always implicitly in the list. For example, suppose that expectation 
      Expect{protected{Fox}} run: Natural -> Natural.
    occurs in package Rabbit. Then this expectation can only be seen by package Fox or Rabbit, or by any subpackage of Fox or Rabbit. When another package imports package Rabbit, it will ignore this expectation if it is not a subpackage of Fox or Rabbit.

    Mode protected indicates visibility only to the current package and its subpackages. For example, if 

      Expect{protected} Go: Natural -> ().
    occurs in package Rabbit, then this expectation of Go is only visible in package Rabbit or another package that imports Rabbit and is a subpackage of Rabbit.

    Protected mode can be used with Let, Define, Bring, Expect and Anticipate declarations. You cannot control which packages see a species or abstraction.

    Private mode

    A definition or expectation can be marked private, indicating that it is visible only in the current package, not to any other package, including subpackages. Use mode private for this.

    Private mode is used in the export part of a package in situations where you need to use something in the export part that you do not want to export. You can, for example, use it in a pattern declaration. For example, 

       Expect{private} unzork: Zork -> Natural. 
   Pattern zork(?x) => 
     Match ?x = unzork target.
   %Pattern
    allows another package to use zork in a pattern (and so implicitly to use unzork as the result of the pattern match) but not to use unzork explicitly.

    Interface and implementation packages

    If you break a package into separate interface and implementation packages, then the implementation part is considered to be part of the overall package. So anything that is marked private in the interface part, for example, is visible in the implementation part.
