    • The concept of a class

    The class is a fundamental concept of object-oriented programming. A class has several important aspects.

    1. A class is a type. This makes it possible to create objects, which are members, or instances, of that type.

    2. A class is an abstraction. This makes it possible to use a class for controlling polymorphism and leads to the concept of inheritance.

    3. A class is a module. So a class can be used to control visibility of definitions.

    Astarte separates these aspects of a class into different concepts: abstractions, species and packages. But a mechanism that combines the first two of them, the type and the abstraction, is available, and that mechanism is discussed here. Modules, and aspects of modules that are related to object-oriented programming, are discussed with the package hierarchy.

    Declaring classes

    In order to use the class feature described here, you must import library package object.ast.

    An object can be thought of as a collection of constants, variables and functions. A class declaration describes those components of an object. A basic class declaration has the form 

      Class name
    component
    ...
    component
  %Class
    where name is the name of the class. This creates a genus called name (the name of the class) and a species called name! (the name of the class followed by an exclamation point). Each component has one of the following forms.

    1. Const const-name: species %Const
      This indicates that each object of this class contains a constant with the given name and the given species. You can declare several constants of the same type as Const a,b,c: species %Const

    2. Var ... %Var
      This defines variables and arrays that belong to an object of this class. All forms of var-declaration can be used.

      You can use the constants that belong to this class as array sizes or as the initial values of variables. Avoid using the value of one variable to initialize another one, since the order in which the variables are created is not defined.

    3. Expect ... %Expect
      This describes expectations that are associated with an instance of this class. These are the methods that an object can perform.

      In object-oriented programming, each operation is thought of as being carried out by some object acting as the agent of that operation. The agent is an implicit parameter of the operation. To take that into account, the expectations that occur inside a class are modified as follows.

      Suppose that the class is called C. Then declaration 

         Expect f: T.
      within this class is interpreted to mean 
         Expect f: C`a -> T.
      This introduces an implicit first parameter that is the agent of the operation.

    4. Anticipate ... %Anticipate
      This describes anticipations that are associated with an instance of this class. These are the virtual functions of the class. These are modified similarly to expectations, except that the added variable is enclosed in braces. For example, if you write 
          Anticipate fun: Natural -> ().
      inside a class called Ghost, the anticipation that is created is 
          Anticipate fun: {Ghost`a} -> Natural -> ().

    Since a class declaration creates a genus and a species, it must occur within an extension declaration.

    An example class

    A class whose instances are stacks of strings can be declared as follows. 
      Extension
    Class StringStack
      Const size : Natural.            %% The physical size of an array that
                                       %% holds the stack contents.

      Var 
        depth! holding Natural := 0;    %% The current number of members
        content(my size) holding String %% The stack contents
      %Var

      Expect
        Push  : String -> ();
        pop   : () -> String;
        Empty : () -> ();
        empty?: () -> Boolean
      %Expect
    %Class
  %Extension
    This creates a genus StringStack (the abstraction of this class) and a species StringStack!. An object of species StringStack! has the attributes described in class StringStack, and no more. The expectations that are created are

    Push: StringStack`a -> String -> ()
    pop: StringStack`a -> () -> String
    Empty: StringStack`a -> () -> ()
    empty?: StringStack`a -> () -> Boolean

    The context of a class

    When a class is defined, a context is created for it, having the same name as the class. The constants, variables and expectations are all put into the context using bring forms. For example, the StringStack class gets an automatic context defined as follows. 
       Context StringStack
     case bring size, depth!, content, Push, pop, Empty, empty?
   %Context
    When inside the context of a StringStack object, you can refer to the parts of that object by putting my in front of the name of the part. For example, my size is the size of the object (a natural number) and my empty? is the empty? function of the object (a function of species () -> Boolean).

    Defining functions for a class

    Functions are defined for a class exactly as if they were not being defined for a class. You must keep the implicit parameter in mind, however. One way to do that is to perform a definition using the context of the class. Here is an implementation of StringStack. Note that these are written outside the class. These definitions use the context StringStack, and assume that you are familiar with contexts. 
       Exception 
     trap stackUnderflowX; 
     trap stackOverflowX
   %Exception

   Define Ask StringStack ?s => empty?() %Ask =
     @(my depth!) == 0
   %Define

   Define Ask StringStack ? => 
            Empty. 
          %Ask =
     Assign my depth! := 0.
   %Define

   Define Ask StringStack ? => pop() %Ask =
     If my empty?() then 
       fail stackUnderflowX
     else
       Let currentIndex = @(my depth!).
       Decrement my depth!.
       Value @( my content # currentIndex ).
     %If
   %Define

   Define Ask StringStack ? => 
            Push(?s). 
          %Ask =
     If @(my depth!) >= my size then
       fail stackOverflowX
     else
       Increment my depth!.
       Assign (my content) # (@my depth!) := s.
     %If
   %Define
    You are not required to use Ask in the definitions, but if you don't, then you must either invoke the context explicitly or forego using the context. For example, definitions 
       Define Empty ?s. = 
     Context StringStack s =>
       Assign my depth! := 0.
     %Context
   %Define
    and 
       Define Empty ?s. = 
     Assign depth!(s) := 0.
   %Define
    are both equivalent to the definition of Empty given above. Choose the form that you prefer.

    Defining constructors

    Although species StringStack! is created as part of the StringStack class, no direct method for creating an object of species StringStack! is provided. Instead, it is up to you to define constructors.

    A constructor definition for class StringStack (really for species StringStack!) has the form 

       Define StringStack constructor
     name params =
        preamble
     then
        postamble
   %Define
    where

    1. The constructor name is name. It is a function that takes some parameters and produces a result of species StringStack!.

    2. Taken together, the function heading (name params) and the preamble must define, for each class constant cname, an identifier my cname (or my-cname. For example, the StringStack class has a constant called size, so an identifier named my size must be defined. This is the value of that constant in the constructed object.

      The preamble is not inside the context of the object being constructed, since it is run before the object is built.

    3. The postamble performs additional initialization of the object, typically storing values into the object's variables. It is implicitly put into the context of the object that is being built, so it can refer to the parts of the object using my ...
    The word then and the postamble can be omitted.

    Here is a constructor for the class StringStack. It has type Natural -> StringStack!. It takes a stack size and produces an object that is a StringStack! with the given size. In this example, both the preamble and the postamble are present. For demonstration only, the postamble puts 0 into the depth! variable. 

      Define StringStack constructor
    stringStack(?my size) =
      ()
    then
      Assign my depth! := 0.
  %Define
    Since the depth! variable is initialized in the class definition, it is not really necessary to assign it a value. Here is an alternative definition of this constructor, omitting the postamble. 
      Define StringStack constructor
    stringStack(?my-size) = ()
  %Define
    To create a new object of class StringStack (really of species StringStack!), you can write 
      Let aStack = stringStack(50).
  Ask aStack => Push("horse"). %Ask
  ...
    Notice that there is nothing that says you must use Ask with objects. You could just as well write 
      Push aStack, "horse".

    The constructors for a class make use of a hidden basic constructor that is declared to be private to the package where the class is defined. Consequently, you can only declare constructors for a class in the same package as the class.

    Subclasses

    Since a class is (partly) an abstraction, it is possible to define new species that belong to that abstraction. Normally, that is done by creating a subclass. You create a subclass by adding isKindOf Superclass after the name of the class, where Superclass is the name of the superclass. For example, you can create a kind of StringClass that has an additional operation, taking the top of the stack without poppping. 
      Class TopStringStack isKindOf StringStack
    Expect top: () -> String.
  %Class
    Declaring a superclass has the following effects.

    1. The context of the class contains not only those things defined in the class, but everything that is in the context of the superclass. The context for class TopStringStack is defined by 
        Context TopStringStack isKindOf StringStack
    case bring top
  %Context

    2. Part of an object of species TopStringStack! is an object of species StringStack!. In order to construct a TopStringStack! object, you must say what this embedded object is. You do this by defining identifier super in the heading or preamble of the constructor.
    The implementation of class TopStringStack might be written as follows. 
      Define TopStringStack constructor
    topStringStack(?my size) =
      Let super = stringStack(my size).
  %Define

  Define Ask TopStringStack ? => top() %Ask =
    If my empty?() then
      fail stackUnderflowX
    else
      @( (my content)  #  @(my depth!) )
    %If
  %Define

    Class Object

    If you do not define a superclass for your class, the superclass is implicitly Object. Class Object is defined in package object.ast.

    When the superclass is Object, you do not need to define super in the constructor. An embedded Object! is automatically created for you.

    Class object provides some fundamental operations on objects, including the following.

    ==
    Two objects are equal just when they are the same object. The equality test does not look, for example, at the values of the constants or the contents of the boxes in an object.

    $
    The default definition of $ for objects produces "(object n)", where n is the object's identity number.

    Parameterized classes

    You can declare a parameterized class. What you get is a community and a family rather than a genus and a species. For example, you can define a general kind of stack as follows. 
      Extension
    Class Stack(`a)
      Const size : Natural.          %% The physical size of an array that
                                     %% holds the stack contents.

      Var 
        depth! holding Natural := 0;  %% The current number of members
        content(my size) holding `a   %% The stack contents
      %Var

      Expect
        Push  : `a -> ();
        pop   : () -> `a;
        Empty : () -> ();
        empty?: () -> Boolean
      %Expect
    %Class
  %Extension
    There is no need to change the implementation except to change the context StringStack to Stack. Type inference will take care of the differences in types.

    This definition of Stack creates an opaque family Stack! and an opaque community Stack. It is usually a good idea to use opaque parameterized classes. If you create a transparent Stack class, for example, then you find that, due to the rules of transparency in the hierarchy, an "object" of species Stack!(Natural) is not really an object at all, since Stack!(Natural) is not beneath genus Object in the hierarchy. Because of this, you lose the basic services provided by class Object.

    Multiple Inheritance

    Sometimes, you want to inherit from more than one superclass. That can be done, but not by using multiple isKindOf phrases. Each class can only have one superclass defined in that way.

    If you want to inherit from more than one superclass, you must do yourself what the class feature does for you. Here are the things to do to make class Sub a subclass of class Sup.

    1. An object of species Sub! needs to be convertible to an object of species Sup!. The operation that does this is called retract, and its species is Sub! -> Sup!. It is the retract function that allows Sub to inherit the variables and constants of Sup!. You must define this function.

      If defining retract is problematic or inefficient, you can alternatively override the selectors for constants and variables that would normally be inherited through the retract function.

    2. Declare genus Sub to be beneath genus Sup, but saying 
          Relate Sub isKindOf Sup.

    3. You probably want to augment the context Sub with things that are inherited from Sup. If you want to inherit everything, write 
          Context Sub isKindOf Sup =>.
      If you want to select what to put in the context, just add what you want. For example, write 
          Context Sub =>
      case bring x,y,z
    %Context

    Classes and modules

    Normally, a class is placed in a module by itself, or along with other things that need to share operations with it. The visibility relations of the package reflect the superclasses of the class. For example, you create a package 
      Package Fox isKindOf Mammal

  export
    Class Fox isKindOf Mammal
      ...
    %Class

  implementation
    ...
  %Package
    Inside the class, anything that is marked private is visible only in this package (and so only to this class, if that is all that is in this package). Anything that is marked protected is visible only to this class and its subclasses (since they, presumably, are in packages that are subpackages of this package).

    Descriptions of classes

    After the heading of a class, and just before listing the components, you can optionally write a string that describes the class. For example, 
      Class Fox isKindOf Mammal
    %" A fox has a bushy tail.
    ...
  %Class
    describes class Fox.

    What is a constant?

    A "constant" in a class can be anything at all. It can be a function, or even a variable. For example, you can define a class Foo by 
       Class Foo
     Const fun: Natural -> Natural.
     ...
   %Class
    Functions that are described by expectations in a class are thought of as belonging to an object, although they are not really stored with the object at all. But a function that is a constant in a class is genuinely stored with the object. That gives you a great deal of flexibility as to what that function does, since you can put any desired function into the object when the object is constructed.

    However, unless you are careful, constant functions will not have access to the other parts of the object to which they belong. You must give them explicit access. To do that, add an extra parameter, as is implicitly done for expectations, and pass the object being constructed to the function when you construct the object. For example, suppose that function funsrc has type Foo`a -> Natural -> Natural. Suppose that you write a constructor for class Foo where my-fun is the first parameter. Then write 

       Define myfoo = foo(funsrc myfoo, ...)
    This recursive definition allows the object to refer to itself.

    Overriding functions

    Normally, definitions made in superclasses are inherited in subclasses. You can override those definitions if you have made them defaults, and you include option override in the overriding declarations in the subclass. For example, if class Mammal is defined by 
      Class Mammal
    ...
    Expect{default} Jump: () -> ().
    ...
  %Class
    then class Fox can override the jump method as follows. 
      Class Fox isKindOf Mammal
    ...
    Expect{override} Jump: () -> ().
    ...
  %Class
    You will need to include an overriding definition of jump in the Fox package. It will begin 
       Define{override} Jump. = 
     ...
   %Define
