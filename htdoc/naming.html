    • Naming things

    You can give things names using let, define, match and relet statements. A name can be used for the remainder of its scope. See scoping, below. For example, 
          Let x = 5.
      (x+x)
    is an expression whose value is 10.

    Forms for binding identifiers

    Let i = e %Let
    This defines identifier i to be the value of expression e. Let statements are eager; e is evaluated immediately, and its value is given a name.

    There are some important restrictions on let statements.

    1. A let statement can only be used in a context where the identifier i being defined currently has no value; it cannot be used to change the meaning of a name.

    2. Let statements cannot be used to make recursive definitions. Use define statements for that.

    Example: Let zebra = "stripes".

    Define i = e %Define
    This defines identifier i to be the value of expression e.

    Define statements are lazy -- the value of e will not be computed until the value of i is used.

    Like a let statement, a define statement can only be used in a context where the identifier i being defined has no value; it cannot rebind a bound identifier.

    A define statement can be recursive. That is, you can define i in terms of itself.

    Example: Define ones = 1 :: ones.

    Relet i = e %Relet
    This redefines identifier i to be the value of expression e. It can only be used in a context where identifier i is already defined; it changes the binding. Moreover, i must be a local identifier in the same innermost function or lazy expression where i is initially defined.

    Relet statements are eager. That is, the value being given a name is evaluated immediately.

    Example: Relet zebra = "black-and-white".

    The following is not allowed, since x is redefined in a lazy context other than the one in which it is initially defined. 

           Let x = 2.
       Define y =
         Relet x = 3.       %% Not allowed.
         (x + x)
       %Define

    Match p = e %Match
    This matches pattern p against the value of e. See pattern matching. Match statements are normally eager. They evaluate e before doing the match.

    Example: Match (?x,?y) = p.

    Scope

    With some exceptions, discussed below, the scope of an identifier definition extends from the point where the identifier is defined to the end of the innermost bracketed construct that contains the definition. A bracketed construct is one that starts with a word W (such as Let) and ends with %W (such as %Let) or with a period or semicolon that stands for %W. For example, in 
            Let a =
           Let b = 2.
	   (b+b)
        %Let
    the scope of b is the interior of the Let a = ... statement. So b cannot be used outside that statement.

    There are some exceptions to this scoping rule.

    1. Ordered pairs. Ordered pairs can be computed in any order. If you write (A,B), where A and B are expressions, then either A or B might be evaluated first. Hence, neither can see bindings done by the other. Bindings done within an ordered pair expression are not visible outside the ordered pair, and bindings done in one component of the ordered pair are not visible in the other component. For example, 
            (Let a = b*b. (a*a), c)
      is allowed, but the binding of a is not visible outside the ordered pair.

    2. Function application. In many programming languages, a function must be an identifier. For example, you can write f(x), where f is an identifier. In Astarte, a function can be an arbitrarily complex expression. The order of evaluation of function applications is also not defined. That is, if you write a(b) where a is an expression that computes a function and b is an expression that is the argument for that function, then either a or b can be computed first. Bindings done in one part of expression a(b) are not visible in the other part. However, bindings done in the parts are visible outside the expression. So 
            ((Let x = 1. Writeln) ([$x]))
      is not allowed, since it violates this scope rule, but 
            ((Let x = 1. Writeln)(["Good"]))
      Writeln[$x].
      is allowed.

    3. Failure. In some situations, an identifier definition becomes unavailable due to a failure. So in 
             Try
         Let a = f(x).
         Let b = g(x).
       then
         ... (a and b are available here.)
       else
         ... (neither a nor b is available here.)
       %Try
      a and b are not available in the else part because, in order to get there, the part that computes a and b must have failed.

    4. Open Expressions. You can place the word open in front of some constructs. Doing so causes definitions made inside them to be visible outside. It opens the scope. Modifier open is most useful for conditionals and loops. For example, if you write 
            If x == 0
         then Let y = 0.
         else Let y = 1/x.
      %If
      the value of y is not available outside the if expression. But if you write 
            open If x == 0
         then Let y = 0.
         else Let y = 1/x.
      %If
      then the value of y is available outside. Only those identifiers that are bound on every possible branch can be used outside.

      The following kinds of expressions support open.

        Ask   Let   Stream
        Atomic   Loop   Trap
        Choose   Mix   Try
        CutHere   Match   Untrap
        Define   (procedure application)   Unique
        If   Relet
      Additionally, you can place the word open in front of an ordered pair or tuple. This makes the bindings done in the pair or tuple visible outside. It also forces the tuple to be evaluated from left to right, so that bindings done in each component are visible in components to the right of it.

    Closing the scope

    If you want to close the scope of a section of code, so that identifier bindings done in it are not visible outside, you can use function CloseScope. CloseScope is just the identity function (CloseScope(x) = x), but since it is a procedure identifier, using it closes the scope. So 
           CloseScope
         Let x = y*y.
	 Writeln[$x].
       %CloseScope
    does not allow the definition of x to be seen outside.

    Mutual recursion

    Define statements can be used to make recursive definitions of functions, but they do not allow mutual recursion, since each identifier must be defined before it is used. To do mutual recursion, use a team. It has the form 
        Team
       definitions
    %Team
    where definitions is a sequence of definitions for the identifiers. Each definition should be a define statement. For example, here is a mutually recursive definition of two functions even and odd. It uses function definitions. 
           Team
         Define even by
           case even(0)    = true
           case even(?n+1) = odd(n)
         %Define

         Define odd by
           case odd(0)    = false
           case odd(?n+1) = even(n)
         %Define
       %Team

    Choosing values to name

    You can use a let or define statement of the form 
          Let x by choice-mode
        case x = a1 when c1
        case x = a2 when c2
        ...
      %Let
    to define an identifier (x in this case). Each of c1, c2, etc. is a condition that must be true for the case to be chosen. The cases are tried according to the choice mode, which must be as in defining functions by cases. Optionally, the last case can begin with else, and have no when clause. For example, if you want to set m to the smaller of u and v, you can write 
          Let m by one
        case m = u  when u <= v
        case m = v  when v <= u
      %Let
    or 
          Let m by one
        case m = u when u <= v
        else m = v
      %Let
