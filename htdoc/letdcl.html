    • Binding identifiers in the outer environment

    The outer environment (or global environment) is a table shared by all parts of a program. A key in the table is a pair consisting of an identifier and a species. The value associated with pair (id,T) must be a value of species T.

    For example, there is a binding in the outer environment of all programs that associates the value 0:Natural with the pair (zero, Natural). So all programs can use the name zero as a name for the number 0.

    Let and define declarations are called binding declarations. They are used to create bindings in the outer environment.

    Binding declarations have forms that are similar to let and define expressions. There are a few important additions, however.

    Polymorphism

    Binding declarations can be polymorphic. For example 
          Define f(?x) = (x,x).
    defines f: `a -> (`a,`a). That is, there is a definition of f for each species T -> (T,T). See type system.

    Overloading and multiple definitions

    Overloading is discussed under expect declarations. If a binding declaration uses an overloaded identifier, then it can lead to more than one definition being made. For example, 
          Define two: Natural = 2.
      Define two: String  = "two".
      Define duo = two.
    provides two different definitions of two. The binding of duo implicitly provides two different definitions of duo, one for each definition of two.

    Occasionally, this causes a binding declaration to produce two conflicting definitions. If this happens, you will need to restrict the types using assumptions or tags. For example, if you write 

          Define two: Natural = 2.
      Define two: String  = "two".
      Define twoname = $(two).
    you get two different definitions of twoname, both of species String. This can be avoided by 
          Define twoname = $(two: Natural).
    if that is what is desired.

    Overriding definitions

    You can override definitions in the outer environment. Use modes default, override and underride.

    A binding declaration marked by mode default can be overridden. A declaration that overrides it must have option override. For example, you can write 

          Define{default} f(?x) = x.
      Define{override} f(?x) = reverse x.
    Notice that the default definition in the above example has a higher amount of polymorphism than the overriding definition. The default definition can be used for any species of parameter at all, but the overriding definition can only be used on lists. So f(2) = 2, by the default definition. But f[2,3,4] = [4,3,2], by the overriding definition, which takes precedence when the parameter is a list.

    You can also make a definition that defers to prior definitions, and only provides cases that earlier definitions did not cover. Mode underride is used for that. So 

          Define{underride} f(?x) = x.
    will provide additional definitions of f, which are used only if prior definitions are not available. The prior definitions are not required to have option override in order to override an underriding definition.

    Type inference

    Type inference is done slightly differently for let declarations as for define declarations.

    1. In a let declaration that defines X, each occurrence of X is assumed to refer to an expectation or definition that was made earlier. Each occurrence of X can have a different species. You can write a recursive definition using let, but only if a prior expectation is given. For example, 
            Expect factorial: Natural -> Natural.
      Let factorial by
        case factorial(0)    = 1
        case factorial(?n+1) = (n+1)*factorial(n)
      %Let
             is allowed. See expect declarations.

    2. In a define declaration of X, each occurrence of X is assumed to have the same species as the X that is being defined. No earlier expectations are consulted for them. So you can write the factorial function without an expect declaration using define. But if you try to define $ for ordered pairs using define, you get into trouble. An obvious definition is 
           Define $(?a,?b) = "(" ++ $(a) ++ "," ++ $(b) ++ ")".
             But this causes a type error, since the uses of $ in the body of the function are not of the same species as the $ that is being defined. You need to use a let declaration for this kind of definition.

    Specifying the species

    You can specify one or more species for the identifier being defined in a binding declaration. Declaration 
         Define species T
       id = val
     %Define
    indicates that id has species T. You can request that more than one species be tried, as in 
         Define species T1
            species T2
       id = val
     %Define
    If there is a mode, it occurs before the word species, as in 
         Define{override} 
         species T1
         species T2
       id = val
     %Define
    Normally, expectations made in the current package are used as tags in a binding declaration (see local expectations). If you use one or more species phrases, then the compiler does not use expectations as tags, but instead uses the explicitly given species as tags.

    Giving a description

    You can provide a description with a binding declaration. Write the word description, followed by the description, after the initial word let or define (or after the mode if there is a mode). For example, 
         Define 
       description "Compute the smaller of a and b"

       min(?a,?b) = If a < b then a else b %If.
     %Define
    You can have both description and species phrases. They can occur in any order.
