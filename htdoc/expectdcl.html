    • Polymorphism and overloading

    Polymorphism in Astarte is via overloading. That is, you can give many bindings of an identifier. Identifier zero, for example, has a binding for species Natural, one for species Integer, one for species Rational and another for species Real (plus definitions for secondary species such as REAL). (In all cases, it names the number 0.) The compiler must keep track of this overloading somehow. Two methods are used.

    1. The different species of an identifier can be collected together into a single polymorphic type called an expectation. This is what is done, for example, with identifier zero. The polymorphic type of zero is REAL`a.

    2. The different species of an identifier can be kept separate. For example, it would be permissible to tell the compiler that zero has definitions of species Natural, Integer, Rational and Real, with no connection between them. In this case, there would be four different expectations for zero. This is not the preferred method, but it is sometimes useful.
    The two ways of keeping track of overloading can be combined. There can be several unrelated polymorphic expectations. For example, function (-) has two expectations, one for the unary negation operator (RRING`a -> RRING`a) and another for the binary subtraction operator ((RRING`a, RRING`a) -> RRING`a).

    Expectations are controlled using expect and anticipate declarations. Such a declaration not only indicates how to keep track of overloading, but it makes a promise that definitions of a particular name will be given for each species in a polymorphic type that is indicated.

    Expect and anticipate declarations also allow you to use what has been promised before the definitions appear, and to separate the promise that the definitions exist from the details of their implementations.

    Expect and anticipate declarations

    Declaration 
           Expect id: T %Expect
    creates an expectation of identifier id, and promises that identifier id will be defined for each species in polymorphic type T. The definitions that fulfill this promise can be given all at once in a single polymorphic definition, or in separate pieces. How you fulfill the expectation does not affect how the compiler keeps track of the expectation.

    For example, the following promises that rover will be defined for species Natural, Integer and INTEGER, but fulfills the definition with three separate definitions. 

      Expect rover: INTEGER`a.
  Define rover: Natural = 2.
  Define rover: Integer = 3.
  Define rover: INTEGER = <<rover:Natural>>.
    Now rover:Integer is 3, but rover:Natural is 2. Of course, this is rather bizarre, but it illustrates that you can have different definitions for different species if you like. Typically, an expectation is fulfilled with just one definition, as in 
      Expect rover: INTEGER`a.
  Define rover = 2.
    which defines rover to be 2 for all three species Natural, Integer and INTEGER.

    Declaration 

      Anticipate id: T %Anticipate
    is similar to an expect declaration, but there are important differences, noted below. Briefly, anticipations are used to introduce virtual functions.

    What do expectations and anticipations promise?

    Anticipate declaration 
            Anticipate id: T.
    promises that

    1. Somewhere in the program (possibly in other packages) definitions of id will be given for every species in polymorphic type T.

    2. These definitions are only guaranteed to cover all types in polymorphic type T if no further extensions are made to data abstractions. That is, the anticipation promises to cover only what exists. Later, when extensions are done, those extensions must also be covered by some definitions of id. Thus, anyone who extends an abstraction might take on the responsibility to provide more definitions of id.
    On the other hand, declaration 
           Expect id: T %Expect
    makes the following promises.

    1. Definitions of id will be given in this package (or in a package that this package imports) for every species in polymorphic type T.

    2. These definitions will continue to cover all species in polymorphic type T even after data abstractions are extended. No other package needs to take on responsibility for providing definitions of id.

    Expectations and type inference

    A local expectation is an expectation (not an anticipation) that was made in the current package, not in a package that was imported.

    When the compiler encounters a let or define declaration 

         Define id = ... %Let
    it looks for local expectations. If there are none, then it allows the most general polymorphic type of id to be defined. But if there is at least one local expectation, then the compiler presumes that this definition is intended to meet at least one of those expectations. The compiler tries the type of each of the local expectations, one after another, as a tag on id. For example, if there are two expectations 
         Expect id: A.
     Expect id: B.
    the compiler first tries 
         Define id:A = ... %Let
    and then tries 
         Define id:B = ... %Let
    Consequently, if you give any expectations for an identifier, you must give all expectations that you intend to satisfy.

    Importing expectations

    When an expect declaration is seen while importing a package P, it is presumed that the expect declaration describes something that is defined in the implementation of P. The implementation itself is not read. So the expect declaration is treated somewhat like the definition that it describes. You can provide modes that are attached to these phantom definitions. For example, 
           Expect{default} id: T.
    indicates that the definition has mode default, and so the phantom declaration will also have mode default.

    This does not happen with an anticipate declaration, since an anticipation does not indicate that anything is defined in the current package.

    If a package gives an anticipation, and partially defines what is anticipated, then its interface should have both an anticipation and an expectation.

    Overloading

    The compiler keeps track of the expectations and anticipations that you have given as a set of polymorphic types. It is required that no two of these types overlap unless one contains the other. For example, it is allowed to say 
          Expect munch: `a -> Natural.
      Expect munch: (`a, Natural) -> Natural.
    since the two types of munch have no species in common. It is also allowed to say 
          Expect crunch: `a -> Natural.
      Expect crunch: Natural -> Natural.
    since one of these types is included in the other. But it is not allowed to write 
          Expect scrunch: `a -> Natural.
      Expect scrunch: Natural -> `a.
    since neither of these types is contained in the other.

    It is only the polymorphic types that the compiler is keeping that must not overlap. In the example of crunch, the compiler only keeps the more general type `a -> Natural. So it is allowed to write 

          Expect crunch: `a -> Natural.
      Expect crunch: RATRING`a -> Natural.
      Expect crunch: INTEGER`a -> Natural.
    since the compiler is only interested in type `a -> Natural. But 
          Expect crunch: RATRING`a -> Natural.
      Expect crunch: INTEGER`a -> Natural.
    is not allowed, since neither of INTEGER and RATRING contains the other.

    Anticipations and virtual functions

    Where an expectation indicates that the current package is taking on some responsibilities, an anticipation puts responsibilities on other packages. Consider the following anticipation. 
          Anticipate f: MyGenus`a -> MyGenus`a.
    This indicates that, if species S is added to MyGenus, then a definition of f: S -> S should be provided. It is usually the responsibility of the package that adds S to MyGenus to provide the definition of f as well. Since function f has not been defined where the anticipation is made, it is called a virtual function.

    When you add a species to a genus, you typically need to do an expectation of the functions that will be provided as part of the obligation from an anticipation. For example, if you add species S to MyGenus, you would need to announce that you will define f. You can write 

          Relate S isKindOf MyGenus.
      Expect f: S -> S.

    You can request that such an expectation be provided automatically when a new species is added to an abstraction. To do that, enclose variables in {...} in the anticipate declaration. For example, 

           Anticipate f: {MyGenus`a} -> {MyGenus`a}.
    indicates that, if species S is added to MyGenus, then declaration 
           Expect f: S -> S.
    should automatically be issued. You do not need to enclose all variables in {...}. If you write 
           Anticipate f: {MyGenus`a} -> MyGenus`a.
    then what is automatically created when species S is added to MyGenus is 
           Expect f: S -> MyGenus`a.
    Generally, any variable that is enclosed in braces is replaced with the newly added species.

    If MyGenus has a subgenus MySubgenus, then adding a species S to MySubgenus implicitly adds S to MyGenus as well. This will be noticed by the compiler, and the above expectation will be generated automatically.

    These automatic expectations have some limitations. An expectation will only be created for you if there are not two different variables in {...} whose domains are being extended. For example, if you write 

           Anticipate f: ({MyGenus`a}, {MyGenus`b}) -> ().
    then no automatic expectation will be created when species S is added to genus MyGenus.

    Suppressing Automatic Expectations

    When you extend an abstraction, normally expectations are automatically created from anticipations, where they have been indicated. If you do not want that to happen, use mode noExpect. For example, you can write 
        Relate{noExpect} MySpecies isKindOf MyGenus.
    This suppresses all expectations that would normally be created from anticipations. If you like, you can selectively suppress expectations. Write 
        Relate{noExpect{f,g,h}} MySpecies isKindOf MyGenus.
    to suppress only expectations for f, g and h, but allow others.

    Adding descriptions to expectations and anticipations

    You can provide a description with an expect or anticipate declaration. Place it immediately after the type. For example, 
           Expect f: T "description of f",
    is equivalent to 
           Expect f: T.
       Description f: T "description of f".

    Reneging on your promises

    At the end of a compilation, the compiler will check to see if all expectations and anticipations have been met. If they have not, the compiler will let you know, for each incomplete expectation, at least one species that is missing from the definitions. Occasionally (but not often), you must give an expectation that promises more than you intend to deliver. You can indicate this using missing declarations, which tell what you intend to leave out.
