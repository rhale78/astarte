D4. Descriptions and Description Declarations

  • Descriptions
  • Embedded descriptions
  • Inconsistent descriptions

    • Descriptions

    You can provide a description of a definition, a species, etc. These descriptions can be obtained by the help facility, so that you can ask for a description.

    Descriptions can be provided where the definitions are made, or separately in description declarations. A description declaration has the form 

        Description 
       thing-being-described description
    %Description
    where description is a string constant and thing-being-described is one of the following.

    identifier : Type
    The description applies to the definition of the given identifier, of the given (polymorphic) type. You can omit :Type if it can be inferred from context.

    Identifier
    You can provide a description of a species, family, genus or community.

    String constant
    You can provide a description of any desired string. Typically, the string is some term that relates to what you are doing.
    Here are some example descriptions. 
          Description
        $         : `a -> String

                  %" $(x) is a string that describes item x.
                  ;

        ANY       "ANY is a genus that contains all species.";

        "horse"

                  %" A horse is a large four-legged mammal.
      %Description

    Embedded descriptions

    Descriptions can be included in some other declarations. For example, a species can be given a description by putting the description at the end of the declaration. You might write 
       Species Complex = (Real, Real)
      " Species Complex holds complex numbers."
   %Species
    The following kinds of declarations can have descriptions with them.
  • Species declarations
  • Expect species declarations
  • Abstraction declarations
  • Expect declarations
  • Let/Define declarations
  • Pattern declarations

    • Inconsistent descriptions

    If you provide more than one description of the same thing, those descriptions must be compatible with one another. If incompatible descriptions are encountered by the compiler, a warning is given. Rules for compatibility are as follows.

    1. Blanks, tabs and newline characters are ignored when performing compatibility tests.

    2. If one description is a prefix of another, then the descriptions are compatible.

    3. If a description contains #{...}, then it is compatible from that point on with any other description that does not contain the exact same sequence #{...}, ignoring blanks, at the same place. For example, string "abc#{frog}def" is compatible with "abc#{toad}xyz", but "rst#{frog}xyz" is not compatible with "abc#{toad}xyz" because those strings differ before the #{...} sequence. This is used to provide descriptions of different natures for the same thing.

    4. Any leading sequence of the form @{w1 w2 w3} is assumed to contain keywords for the description, and is ignored when doing compatibility tests.
