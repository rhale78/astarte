    Logic programming

    There is more than one kind of logic programming. The kind described here is representative of the language Prolog.

    Like functional programming, logic programming is based on writing facts. However, the facts are written in a different form. Where in functional programming the facts are equations, in logic programming the facts are implications. That is, you say that one thing implies another.

    Predicates

    In order to use logic programming, we need the notion of a predicate. A predicate is a function that produces a Boolean result. So it classifies things into just two groups: true and false. An example is predicate odd?, where odd?(1) = true but odd?(2) = false.

    In logic programming, it is convenient to use a different form of predicate. Instead of producing a Boolean result, a predicate produces a result of species (). That is, it produces no useful information. However, the predicate succeeds when it is true, and fails when it is false. The standard predicate odd? (which produces a Boolean result) can be made into a funtion that fails on an even number as follows. 

      Define Odd(?x). = {odd?(x)}.
    In this style, the constant () is thought of as standing for true, and {false} stands for false.

    Stating facts

    Facts are implications, saying that one thing is true if another is true. For example, suppose that Mother(x,y) succeeds if x is the mother of y, and we want Parent(x,y) to succeed if x is a parent of y. We should certainly state the fact that, if Mother(x,y) is true, then Parent(x,y) is also true.

    Implications are written backwards. To say Mother(x,y) implies Parent(x,y), you write 

       case Parent(?x,?y). <- Mother(x,y).

    Just as a function in a functional program can have several defining equations, a predicate in logic programming can have several defining facts. Suppose that Father(x,y) succeeds just when x is the father of y. Then here is a definition of the Parent predicate. 

      Define Parent by all
    case Parent(?x,?y). <- Mother(x,y).
    case Parent(?x,?y). <- Father(x,y).
  %Define

    Basic facts

    A program needs to include some facts that are not implications. They can be stated by saying that they are implied by true, which is written () in a logic program. For example, facts about predicates Mother and Father can be stated as follows. 
      Define Mother by all
    case Mother("minnie", "joan"). <- ()
    case Mother("joan", "curly").  <- ()
    case Mother("joan", "larry").  <- ()
  %Define

  Define Father by all
    case Father("larry", "mike"). <- ()
  %Define

    Computation

    Computation is performed using two key tools.

    1. Choice mode all indicates that backtracking should be used to try the cases. For example, when you try to compute Parent(x,y), two threads are created. The first thread tries to compute Mother(x,y), and the second tries to compute Father(x,y). Backtracking is designed to suspend the second thread until the first one fails. So computation will proceed immediately with Mother(x,y). But if and when that fails, the computation will resume automatically with the other thread.

    2. Logic programming uses unknowns to represent unknown values that are found by the process of unification during computation. Unknowns are explored now.

    How would you define predicate Grandparent, where Grandparent(x,y) is true if x is a grandparent of y? Clearly, Grandparent(x,y) is true (succeeds) if there is some z such that Parent(x,z) succeeds and Parent(z,y) succeeds. But how can z be determined?

    The answer is to let the system find z. Just leave it unknown. When you create an unknown on the right-hand side of <-, you are implicitly asking if that unknown exists. Here is a definition of the Grandparent predicate. 

      Define Grandparent by all
    case Grandparent(?x,?y). <- Var{logic} z.
                                Parent(x,z).
				Parent(z,y).
  %Define
    When you create an unknown, make it a logic unknown, as shown for z.

    Since predicate Grandparent has only one case, it can be written as a single case, as follows. 

      Define Grandparent(?x,?y). <- 
    Var{logic} z.
    Parent(x,z).
    Parent(z,y).
  %Define

    Computation involves solving equations just as for functional programming. However, now the equation solver binds not only pattern variables but also unknowns. For example, suppose that predicates Mother and Father are defined as they are above. Computation of Grandparent("joan", "mike") goes as follows.

    1. There only one fact about Grandparent to use. It says to create a new unknown z, and then to do Parent("joan",z) followed by Parent(z,"mike").

    2. Computation of Parent("joan",z) causes two threads to be tried. The first tries Mother("joan", z), the second tries Father("joan", z). The Mother thread is tried first.

    3. Computation of Mother("joan",z) leads to three threads, for the three cases. The first thread tries to solve equation 
       ("joan",z) = ("minnie","joan")
      There is no solution, so this thread fails. The second thread tries to solve equation 
       ("joan",z) = ("joan","curly")
      This equation succeeds, binding unknown z to "curly". This thread will continue, having finished with Parent("joan", z), and will try to prove Parent(z,"mike"). Note that z is no longer unknown. It has been bound to "curly". Computation of logic programs is active. It binds unknowns to try to make the computation succeed.

    4. The thread that has bound z to "curly" will fail, since we have no information about "curly" being anyone's parent. This will cause the program to backtrack and to try the third thread for Mother("joan", z). That thread tries to solve equation 
      ("joan", z) = ("joan", "larry")
      This equation is solved by binding z to "larry".

    5. Now we return to computing Parent(z,"mike"), with z bound to "larry". That is, we are trying to compute Parent("larry", "mike"). Two treads will be tried, one doing Mother("larry", "mike") (and failing), the other doing Father("larry", "mike") (and succeeding). When all is done, computation of Grandparent("joan", "mike") succeeds, meaning that it is true.
    This process is much easier to see in a tree diagram showing the threads and whether they succeed or fail. Here is a diagram of the computation of Grandparent("joan", "mike"). Each node of the tree contains a list of statements that must be evaluated. The diagram is made more compact by using the abbreviations J = "joan", M = "mike", C = "curly" and L = "larry". The successful thread is shown in blue. 
                               Grandparent(J, M)
                                  |
                         (Parent(J,z). Parent(z,M).)
                        /                          \
  (Mother(J, z). Parent(z,M).)                    (Father(J,z). Parent(z,M).)
  /           |                \                      |
  |           |                 \                   .....
 fail       z = C               z = L
              |                   |
              |                   |
         Parent(C,M).          Parent(L, M).
        /           \          /           \
   Mother(C,M)  Father(C,M)  Mother(L,M)   Father(L,M)
   /   |     \      |        /   |    \       |
fail  fail fail    fail   fail fail  fail   success

    Unknowns in goals

    Logic programming allows you to try to compute, or prove, goals that contain unknowns. Suppose, for example, that you compute 
       Var{logic} who.
   Grandparent("joan", who).
    Computation of this leads to just one successful thread. That thread binds who = "larry". Similarly, computing 
       Var{logic} who.
   Grandparent(who, "larry").
    leads to a single successful thread that binds who to "joan". Try writing the computation trees to see this.

    Recursion

    You can use recursion in definitions. For example, here is a definition of predicate Ancestor, where Ancestor(x,y) is true if x is an ancestor of y. 
      Define Ancestor by
    case Ancestor(?x,?y). <- Parent(x,y).
    case Ancestor(?x,?y). <- Var{logic} z.
                             Parent(x,z).
                             Ancestor(z,y).
  %Define

    Symbols

    In symbolic computing, you often use names for things in the domain of discourse. We have used names "joan" and "larry", for example. If you use strings for those names, you find your program doing frequent equality tests between those strings. Equality testing involves testing character by character.

    The equality tests can be sped up by using symbols. If s is a string the #s is the symbol whose name is s. Symbols have the advantage that equality testing is cheap for them. (The strings that name symbols are stored in a hash table, so that all occurrences of a given symbol point to the same string in memory. Comparing symbols for equality is done by comparing the addresses where their names are stored, not by looking at the individual characters in the names.) If you use symbols, the definition of the Mother predicate looks like this. 

      Define Mother by all
    case Mother(#"minnie", #"joan"). <- ()
    case Mother(#"joan", #"curly").  <- ()
    case Mother(#"joan", #"larry").  <- ()
  %Define
    Alternatively, you can give names to frequently used symbols, and refer to them by their names. For example, you can write 
       Define 
     minnie = #"minnie";
     joan   = #"joan";
     curly  = #"curly";
     larry  = #"larry"
   %Define
    and then write 
      Define Mother by all
    case Mother(minnie, joan). <- ()
    case Mother(joan, curly).  <- ()
    case Mother(joan, larry).  <- ()
  %Define

    Unification

    When a logic program is run, you want equality tests to be active. That is, they should not just test for equality, but should try to solve an equation, binding unknowns to try to make the equation true. Active equality testing is called unification.

    When a pattern variable occurs two or more times in a pattern, the pattern matcher must implicitly do an equality test. For logic programming, you want that equality test to be done using unification. If you use the form of definition that we have used here, using <- in the cases, all equality testing that is called for in pattern matching is done using unification.

    Logic programming with lists

    Logic programming works well for computing with lists. For example, we can define a predicate Append(x,y,z), which is true just when z = x ++ y. Here is its definition. Check each case to see why it is true. 
      Define Append by all
    case Append(nil,    ?a, ?a).     <- ()
    case Append(?h::?t, ?a, ?h::?r). <- Append(t, a, r).
  %Define
    The append predicate can be used in several different modes. For example, to find the result of appending x and y, you can write 
      Var{logic} result.
  Append(x,y,result).
    Suppose you want to compute result = x _lminus y. This is the result of removing list y from the front of list x. So [a,b,c,d] _lminus_ [a,b] is [c,d]. Here is a definition of _lminus_ using the Append predicate. Notice that this definition melds functional and logic programming styles. It uses a predicate defined in a logic programming style to implement a function. 
      Define ?x _lminus_ ?y =
    Var{logic} result.
    Append(y, result, x).
    Value result.
  %Define
    This just says, find me a list called result such that Append(y, result, x) is true. That is, x = y ++ result.

    A membership testing predicate is also easy to write using Append. In this case, we are staying within the logic programming paradigm. Suppose Member(x,y) is to succeed just when x is a member of list y. 

      Define Member(?x,?y). <- 
    Var{logic} a,b.
    Append(a, x::b, y).
  %Define
    This just says that x is a member of list y just when there exist lists a and b such that a ++ (x::b) = y.

    Positive thinking

    When doing logic programming, think in terms of what is true, not in terms of what is false. Anything that you don't say is presumed to be false. For example, here is a direct definition of the Member predicate. 
      Define Member by all
    case Member(?x, ?x::?y). <- ()
    case Member(?x, ?a::?y). <- Member(x,y).
  %Define
    Here is a definition of a member function in a functional style for contrast. Notice that there are three cases, one stating negative information. There is no corresponding case in the prior definition, because in logic programming you only state positive facts. 
      Define member by first
    case member(?x,[])     = false
    case member(?x,?x::?y) = true
    case member(?x,?a::?y) = member(x,y)
  %Define

    Cutting the proof tree

    Sometimes it is necessary to control the computation of a logic program. One way to do that is in careful ordering of your implications, and of the subgoals on the right-hand side of the implications. Another important way of controlling computation is via cuts. A cut destroys other threads, and works as follows.

    1. First, you set a cut point in a thread.

    2. After computation has progressed for a while, and several threads have possibly been created, one of those threads performs a Cut operation. The effect of the cut is to destroy all other threads back to the nearest cut point.
    For example, suppose the computation tree looks like this. Goals have been replaced by the word "goal", and a cut-point is shown. 
                              goal
                         /    \
                     goal      goal
                      |
                   cut-point
                      |
                     goal
                    /    \
                 goal     goal
                /    \
             goal     goal
              |
             Cut
    when the Cut is reached, all other threads (and their associated goals) are removed, back to the cut point, yielding 
                              goal
                         /    \
                     goal      goal
                      |
                   cut-point
                      |
                     goal
                    /
                 goal
                /
             goal
              |
             Cut
    To set a cut point, use CutHere ... %CutHere. The cut point is installed when the CutHere construct is entered, and is removed when that construct is exited. You can wrap CutHere ... %CutHere around the cases in a definition by cases. For example, here is a definition of the Member predicate that does a cut when a success is reached. 
      Define Member by all
    CutHere
      case Member(?x, ?x::?y). <- Cut.
      case Member(?x, ?a::?y). <- Member(x,y).
    %CutHere
  %Define
    The effect of the Cut is to prevent the second case from being tried, since the thread that would have done that case is destroyed.

    Cuts are an important way to decrease memory and time utilization by logic programs. But be careful with them. It takes careful reasoning to understand that a cut is the right thing to do. Arguments in favor of cuts are often wrong in practice.

    Or-parallel logic programming

    You can cause different threads to be tried in parallel by using mode all mixed instead of all. Here is a definition of the member predicate that tries different members in parallel. 
      Define Member by all mixed
    case Member(?x, ?x::?y). <- ()
    case Member(?x, ?a::?y). <- Member(x,y).
  %Define
    This kind of programming is called or-parallel because the alternative ways of doing something (the branches of an or) are tried in parallel.

    Be careful with this. Although it might seem appealing to try things in parallel, doing so can be very costly in terms of memory, since the different branches of the computation are all using memory simultaneously. It is easy to run out of memory in a parallel search.
