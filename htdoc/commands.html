    You will need to have the directories containing the binaries (astc, astr and asthelp) in your path. See setting up for how to set up your path.

    Compiling a program

    There are two ways to compile your program. You can either use the compiler directly or you can use the interpreter; it will compile files as needed. You can jump ahead to the interpreter if you prefer to use that method. (Beginning users: this is the recommended approach.)

    The compiler is called astc. Name your program with extension .ast. When you compile myprog.ast, the compiler will produce file myprog.aso, holding byte-code for the program. To compile myprog.ast, use 

          astc myprog

    Most used options for astc

    The following options are supported. Put them before the name of the program, as in 
          astc -l myprog
    Most options have a short form beginning with a single dash and a long form beginning with two dashes. Some options have only one form.

    --help
    Print a synopsis of the command options. If this option is used, nothing is done except printing the synopsis.

    -i, --no-check-indent
    Suppress indentation warnings.

    -k, --no-check-exhaustive
    Suppress warnings about patterns not being exhaustive.

    -l, --no-listing
    Do not print a listing. (This option is dash-ell.) Normally, a listing is written to standard output.

    -s, --show
    Cause the results of type inference to be dumped when there is a type error. This can help you understand why a type error has occurred.

    -1
    Stop at the first declaration that causes an error. (This option is dash-one.)

    Less used options for astc

    -a, --report-imports
    Show reports of definitions, including those made in imported packages, but excluding those done in the standard package. For example, if a function f: Natural -> Natural is defined, a report of this will be placed in the listing. See -A and -d.

    -A, --Report-imports
    Show reports of definitions, including those made in imported packages, and including those done in the standard package. For example, if a function f: Natural -> Natural is defined, a report of this will be placed in the listing. See -a and -d.

    -b, --behavior
    Turn on behavior checking. This is currently not recommended, since behavior checking is not fully implemented.

    -c, --start-in-comment
    Start reading the package as if in a long comment initially. The package will actually begin at the first occurrence of >>>>.

    -d, --abbreviated-listing
    Suppress a listing, as for -l, but show all reported definitions made in this package. For example, if a function ghost of type Natural -> Natural is defined, a line of the form 
       -------> ghost : Natural -> Natural
    is printed. See -a and -A.

    -e, --expert
    Use short error messages. Normally, longer, more descriptive error messages are used.

    -e<file>
    Append all error messages to file <file>. These are written in addition to the error messages in the listing, if a listing is written.

    -E<file>
    Write all error messages to file <file>, emptying that file first. These are written in addition to the error messages in the listing, if a listing is written.

    -f, --full-listing
    Do a full listing. This is used when compiling a package that has separate interface and implementation parts. Normally the compiler will only list the implementation part, when compiling the implementation. Option -f causes both interface and implementation to be listed.

    -F, --Full-listing
    Do a very full listing. This causes all imported files, including the standard package, to be listed the first time they are encountered. This listing can be quite large.

    -I<dir>, --import<dir>
    Check directory <dir> when looking for imports, in addition to those already being searched. This directory is added to the front of the search path.

    -L<file>, --listing<file>
    Produce listing in the given file. Normally, the listing is to the standard output.

    -ns
    This option should only be used to compile standard.ast. It suppresses the normal automatic import of standard.ast.

    -nw, --no-warn
    Suppress all warnings.

    -pN
    Pattern and expand rules are kinds of macros, and involve doing expansions. Normally, expansions are cut off if too deep, to catch infinite expansions. This option allow pattern match/expand substitutions to continue to depth N.

    -w, --warn
    Override any suppressed warnings. Give the warnings anyway.

    Directories to search

    The compiler will normally look in the current directory and in the standard library directory for imports. You may add a directory to import using the -I option. You can also alter the initial directories to search by setting environment variable AST_IMPORTS to a colon-separated list of directories. You can use $ to stand for the directory that holds the standard Astarte library. For example, you can write C-shell command 
        setenv AST_IMPORTS .:~/astarte:$
    Be sure to include "." in that list to search the current directory.

    Running a program

    After you have compiled a program, you can run it using command astr. In fact, if a file has not been compiled since it was updated, and you try to run it, the interpreter will ask you if you want to compile it. To run myprog.ast (really the compiled version myprog.aso), use 
          astr myprog
    Beginning users: there is no need to read the rest of this page. It only describes further options for running programs.

    Normally, astr looks for file myprog.aso (the compiled version of myprog.ast) in the current directory. You can force astr to look in other directories by setting environment variable AST_PATH to a colon-separated list of directories to search. Be sure to include "." if you want the current directory searched.

    You can include options before the name of the program. Anything that occurs after the name of the program is considered to be a command line argument to myprog, and is accessible to myprog via string commandLine. For example, if you write 

          astr -t myprog -f
    then option -t is taken as an option to astr, and myprog is run with commandLine having value ["myprog", "-f"]. The options for astr are as follows. Most options have a short name that starts with a single dash and a long name that starts with two dashes. Some options have only one name.

    Most used options for astr

    -d, --debug-at-trap
    Start the debugger any time an exception is trapped. See option -v.

    --help
    Print a synopsis of the command options. If this option is used, nothing is done except printing the synopsis.

    -i<file>, --stdin<file>
    Redirect the standard input to the named file. This is more effective than having the operating system do the job, and allows the program still to communicate with you via the keyboard when necessary.

    You can have a space after -i if desired, but none is required. For example, write 

         astr -imydata myprog
    or 
         astr -i mydata myprog

    -m, --make
    Compile files that are out of date without asking for permission.

    -o<file>, -o!<file>, --stdout<file>, --stdout!<file>
    Redirect the standard output to the named file. This is more effective than having the operating system do the job. Forms without ! will ask you whether you want to overwrite an existing file, if the file already exists. Forms with ! will overwrite files without asking. You will need to escape the ! (write \!).

    You can have a space after -o if desired, but none is required. For example, you can write 

      astr -o\!myout myprog
    or 
      astr -o\! myout myprog

    -t, --no-optimize
    Normally, tail recursion is handled without generating a new stack frame. Also, when an identifier binding is no longer accessible to the program, it is deleted, possibly before it goes out of scope. This can be confusing when you are debugging a program. Option -t suppresses both of these behaviors. So with -t, tail recursion is treated as normal recursion and bindings are not removed until they go out of scope.

    -v, --debug
    Enter the debugger to start. Use this to set breaks and see what the program is doing. Type help to the debugger to see debugging commands that are available.

    Less used options for astr

    -e<file>, -e!<file>, --stderr<file>, --stderr!<file>
    Redirect the standard error output to the named file. This is more effective than having the operating system do the job. (Some stderr messages will not be redirected.) Forms without ! will ask you whether you want to overwrite an existing file, if the file already exists. Forms with ! will overwrite files without asking. You will need to escape the ! (write \!).

    -f, --no-check
    Do not query about out-of-date files. Load silently, and use the existing .aso files wherever possible.

    -hN, -h+, --heap-limitN, --heap-limit+
    Normally, there is a soft limit on the size of the heap. If the program uses more than this number of bytes, the user is asked whether the program should continue. Option -hN sets the soft limit on the size of the heap to N kilobytes. Option -h+ removes the soft limit entirely. The default limit is about 1MB. For example, to set the heap limit to 2MB, use 
       astr -h2000 myprog

    -n, --no-run
    Only load and link. Do not run.

    -pN, --precisionN
    Set the initial value of box precision! to N. For example, use 
                astr -p100 myprog
    to cause floating point arithmetic to be done with 100 digits of precision by default.

    --profile
    Produce a crude execution profile.

    -q, --quick-load
    Quick load. Suppress tests.

    -rtN, --dump-sizeN
    When a program traps an exception, a dump of the configuration is printed. The dump is limited in size to about one megabyte by default. Use option -rtN to set the limit to N kilobytes.

    -riN, --dump-value-sizeN
    When a program traps an exception, a dump of the configuration is printed. The display of each item is limited in size by default to 500 bytes. Use option -riN to set the limit to N bytes.

    -sN, -s+, --stack-sizeN, --stack-size+
    Normally, there is a soft limit on the number of stack frames held by a single thread. If a thread uses more than this number of stack frames, the user is asked whether the program should continue. Option -sN sets the soft limit on the number of stack frames to N bytes. For example, use 
                astr -s1000 myprog
    sets the soft limit to 1000 frames. Option -s+ removes the soft limit entirely. The default limit is 400 frames.
