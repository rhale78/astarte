D10. Bring Declarations and Expressions

    • Bring definitions to new species

    You can define a new species New to have representation species Rep by 
        Extension
      Species New = Rep.
    %Extension
    When you do this, you only get a few basic operations for species New. For example, if you write 
        Extension
      Species Distance = Real.
    %Extension
    you define a new species Distance whose values are represented by real numbers. You get a constructor called distance and a destructor called undistance. You do not automatically inherit the ability to add or multiply distance values.

    You can let your new species inherit some operations from the representation species using a bring declaration. For example, to create a function +: (Distance, Distance) -> Distance that adds two distances, you can write 

        Bring +: ({Distance}, {Distance}) -> {Distance}.
    The new definition of + is defined in terms of the old one where each occurrence of a species in braces is replaced by the appropriate representation species. Since the representation species of Distance is Real, + for distances is defined as the same as +:(Real, Real) -> Real, where computation is performed on the representation.

    The type is required in a bring declaration. You cannot, for example, just write Bring + %Bring.

    Bringing several items

    You can bring several items in one declaration or expression, as long as all have the same type. For example, you can bring both addition and subtraction of distances as follows. 
        Bring +,-: ({Distance}, {Distance}) -> {Distance}.

    Renaming

    Typically, you bring a function from another one of the same name. If you are defining just one item, you can change its name. For example, suppose that you want to be able to add distances, but you want the addition function to be called distanceSum. Then you can write 
        Bring distanceSum: ({Distance}, {Distance}) -> {Distance} from +.
    where the from phrase tells the function that is being inherited, but the bring tells the name under which it will be inherited.

    More complex brings

    Bring declarations can switch between objects and their representations in relatively complex ways. For example, suppose that you have a list of real numbers and you want to convert it into a list of distances. You can write a conversion function for that. In terms of the representations, the two lists are the same. So the function that implements the conversion is the identity function. You write 
        Bring asDistances: [Real] -> [{Distance}] from idf.
    This says that function asDistances: [Real] -> [Distance] is implemented, on representations, by idf: [Real] -> [Real].

    The efficiency of brings

    A bring does not introduce any new computation. For example, function asDistances is implemented exactly as idf. The computation is simply done on the representations, and no additional type conversion functions are run. This makes brings very efficient for some operations.

    Allowed conversions

    We say that a conversion between species A and B is allowed if you can write {A} in a bring declaration, and expect {A} to be replaced by B in the type of the function that is inherited, or the other way around.

    1. You can convert between a species and its representation species, as long as the species has just one constructor. You can do this in any context in a type. If there are two or more constructors, you cannot convert.

    2. You can convert between a variable whose domain is RANKED or ENUMERATED and its representation, Natural, provided the conversion occurs in a left context.

    Modes for brings

    You can use the same modes for a bring declarations as for a define declarations. For example, a bring can be a default, or can override a prior definition.
