    • Requesting lazy evaluation

    Normally, evaluation is strict, or eager. That is, expressions are evaluated when they are encountered. You can request lazy evaluation of particular expressions. When you ask for lazy evaluation, evaluation of an expression is deferred until the value of that expression is needed.

    (:e:)
    (:e:) is the same as expression e, but is evaluated on need. Evaluation of (:e:) yields a special item called a promise holding within it a function that will compute the value of e when it is needed.

    Example: (:f(x):) has the same value as f(x), but is not computed until it is needed.

    Be careful about putting commands that have side-effects inside smiles. Those side-effects will not happen until the expression is evaluated. For example, 

       Let x = (:Writeln["I am evaluating x."]. 5:).
   Let z = x.
   Writeln["Now I will evaluate z + 4."].
   Let y = z + 4.
    prints 
       Now I will evaluate z + 4.
   I am evaluating x.
    since the value of x is only computed where it is needed to form the sum z + 4. Notice that just giving the promise named by x another name (z) does not force it to be evaluated.

    (:& e:)
    Normally, whenever a promise is evaluated, its result is memoized. That is, the answer is remembered so that the promise will not need to be evaluated more than once. For example, 
       Let x = (:Writeln["I am evaluating x."]. 5:).
   Let z = x.
   Writeln["Now I will evaluate z + 4."].
   Let y = z + 4.
   Let r = x + 9.
   Writeln["Now I have y and r."].
    prints 
       Now I will evaluate z + 4.
   I am evaluating x.
   Now I have y and r.
    since the promise named by x is only evaluated once.

    Expression (:&e:) requests that memoization be suppressed. This is used when you prefer not to use the memory needed to keep the answer around, and would rather recompute. Whether the system honors this request is up to the system. In some cases, memoization will still be done. In some cases, a memoized value might be held for a while until it is collected by the garbage collector.

    Await then e %Await
    Same as (:e:).

    Example: 

        Await then
      f(x)
    %Await

    Await x then e %Await
    Creating and evaluating promises has some overhead cost associated with it, and it is a good idea to avoid it when it is not necessary.

    This await-expression examines x. If x is something that has not yet been computed, then this await-expression behaves like (:e:); it builds a promise. If x has been computed, and is available, then usually the await-expression evaluates e immediately. Occasionally, it chooses to be lazy. This allows you, for example, to cut off deep recursion by becoming lazy after several recursive calls. This kind of evaluation is called quasi-lazy evaluation.

    Note: When deciding whether a value is known, the system only looks at the top level. If, for example, you have a list that is known to be nonnull, and to have head h and tail t, then that list is considered already evaluated by an await-expression, even if h and t are not evaluated yet. As long as a little bit of the structure of the result is explicitly available, the value is considered evaluated.

    Example: 

        Await x then
      f(x)
    %Await

    Await a,b,c then e %Await
    This abbreviates Await a then Await b then Await c then e %Await %Await %Await. You can wait on any number of things.

    Await& ... then e %Await
    Like (:e:), Await expressions normally memoize their results. Placing & after Await requests that memoization not be done. The system might and might not honor this request.

    Lazy evaluation is described in more detail in lang/lazy.ast.

    Lazy functions

    Some of the library functions are lazy, and produce deferred values when they are run. For example, function lazyMap is defined so that (lazyMap f [a,b,c,d]) = [f(a), f(b), f(c), f(d)]. However, only as much of the result list is evaluated as is needed. If you only evaluate the head of this list, then only f(a) will be computed; f(b), f(c) and f(d) will remain unevaluated.

    Some library functions are quasi-lazy. They are usually eager on structures that are known, but always lazy on structures that are not known. For example, (map f [a,b,c,d]) = [f(a), f(b), f(c), f(d)], so map computes the same function as lazyMap. But map is quasi-lazy. It might compute a little more of the list than you ever look at. Although this can lead to wasted work, it is typically a little faster than using a fully lazy function, because it avoids the overhead associated with creating and then evaluating promises.

    Defining lazy functions

    You can define lazy functions by direct means or using definition by cases. Here is a definition of the lazyMap function without using definition by cases. Instead, we use a choose expression. 
        Define lazyMap ?f ?x =
      Await then
        Choose first matching x
          case []        => []
          case [?h & ?t] => [(:f x:) & lazyMap f t]
        %Choose
      %Await
    %Define

    In a definition by cases, you can wrap the cases in an await expression. Here is an alternative definition of lazyMap. 

        Define lazyMap by
      Await then
        case lazyMap ?  []        = []
        case lazyMap ?f [?h & ?t] = [(:f x:) & lazyMap f t]
      %Await
    %Define

    Defining quasi-lazy functions

    Quasi-lazy functions can be defined similarly. Here is a definition of the map function by cases. 
        Define map ? ?x by
      Await x then
        case map ?  []        = []
        case map ?f [?h & ?t] = [f x & map f t]
      %Await
    %Define

    Library functions

    There are some library functions that are lazy versions of eager library functions. Here are some.

    lazy
    Function lazy takes another function f and produces a lazy version of f. It is define by 
        Let lazy ?f ?x = (:f x:).

    lazy?
    lazy?(x) is true if x has not yet been computed. See notLazy?, below.

    lazyHead
    lazyHead(x) = head(x), but x is not evaluated until the result of lazyHead(x) is needed.

    lazyLeft
    lazyLeft(x) = left(x), but x is not evaluated until the result of lazyLeft(x) is needed.

    lazyRight
    lazyRight(x) = right(x), but x is not evaluated until the result of lazyRight(x) is needed.

    lazyTail
    lazyTail(x) = tail(x), but x is not evaluated until the result of lazyTail(x) is needed.

    notLazy?
    notLazy?(x) is true if x is not lazy at its top level. (For example, a list that is known to be nonempty is not lazy at its top level. An ordered pair is not lazy if expressions for its two parts have been obtained, but not necessarily evaluated. So, for example, if p is an ordered pair that is not lazy, then left(p) cannot force any lazy evaluation.)

    Lazy pattern matching

    The following pattern form is available.

    (:?x:)
    This pattern matches ?x against the target, but it attempts to avoid evaluating the target. For example, 
         Match (: (?a,?b) :) = p.
    has the same effect as 
         Let a = lazyLeft(p).
     Let b = lazyRight(p).
    so that the target p will not be evaluated during the match.

    Forcing evaluation

    Normally, deferred values are computed automatically when they are needed. Occasionally, you want to force them to be evaluated early. Use function force to force evaluation until the value does not appear to be deferred at its top level. Use forceAll to fully evaluate. For example, function pack is occasionally lazy. Writing 
          Let x = force(pack a).
    will force pack to be eager.

    Sequencing and lazy evaluation

    It is possible to create a value of species () that is not fully evaluated. If such a value is used in sequencing, it is not evaluated at all. For example, 
           (:Writeln["First string"].:)
       Writeln["Second string"].
    only prints Second string. The first deferred value is not evaluated. If you want to force evaluation of such an expression, use function force.
