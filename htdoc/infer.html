T9. Type Inference

  • Controlling type inference
  • Monomorphism restrictions
  • Defaults
  • Type inference and recursion
  • Error detection
    • The compiler performs type inference. Typically, you do not need to provide any type information in the program. The compiler will choose the most general polymorphic type that is consistent with each definition.

    Controlling type inference

    You can control type inference using tags and assumptions. Assumptions can be local, within a declaration, or can be declarations themselves. For example, declaration 

          Assume n:Natural.
    indicates that each occurrence of identifier n should be presumed to have species Natural. If the type in an assumption has variables, then those variables are copied at each use. For example, 
          Assume x: [`a].
    indicates that each occurrence of identifier x should be presumed to be a list. A different copy of variable `a is used at each occurrence of x, so that the occurrences of x need not be the same kind of list.

    Monomorphism restriction

    Each local identifier within a declaration is presumed to have the same species at each occurrence. You cannot declare a local identifier and use it with more than one species. So 
          Execute
        Let f(?x) = x.
        Let a = f("abc").
        Let b = f(2).
      %Execute
    is not allowed, since the two occurrences of f have different species. Globally defined indentifiers can be used with different species, however. So 
          Let f(?x) = x.
      Execute
        Let a = f("abc").
        Let b = f(2).
      %Execute
    is allowed.

    Defaults

    Due to the monomorphism restriction, it is sometimes necesssary for the type inference mechanism to supply some information that is not present in a program. For example, suppose that you write the following execute declaration. 
        Execute
      Writeln[$ []].
    %Execute
    The constant [] is polymorphic. It can be any kind of list at all. The compiler assigns it a type of [`a]. But the type of [] cannot be left polymorphic. What the compiler does is select a default species. Unless you say otherwise, a variable whose domain is ANY defaults to species (). So you end up printing [] of species [()].

    You can control how defaults are selected. See default declarations.

    Type inference and recursion

    When you make a recursive definition as a declaration, the compiler needs to know whether you intend for the embedded (recursive) uses of the identifier being defined to have the same species or possibly different species.

    If you use a define declaration, then the compiler assumes that all recursive uses have the same species as the identifier being defined. So it treats the identifier being defined as if it is a locally defined identifier. For example, the declaration 

          Define factorial (?n) =
        If n == 0 
          then 1
          else n * factorial(n--1)
        %If
      %Define
    indicates that the two occurrences of factorial (one in the heading, one in the body) must have the same species.

    A let declaration indicates that each embedded occurrence of the identifier being defined should be treated like any other globally defined identifier, and should get its expectations from prior expectations, anticipations and definitions. For example, here is a definition of $ for ordered pairs. Notice that the uses of $ do not have the same species as the definition of $. For example, if you use this definition with $ having species (Natural, Integer) -> String, then the two uses of $ have species Natural -> String and Integer -> String, respectively. 

          Let $(?a,?b) = "(" $(a) ++ "," ++ $(b) ++ ")".

    Error detection

    If your program cannot be well-typed, the type inference mechanism will give an error. See understanding type errors for a discussion of how to understand the error reports.
