    • Forking threads

    The stream and mix constructs provide two mechanisms for creating threads. Stream creates a thread that is similar to a thread created by a try-expression; it waits to run until another thread has failed. Mix creates a thread that does not wait, but runs immediately.

    Stream a then b %Stream
    This expression forks the current thread into two. One thread evaluates a. It continues running. The other thread evaluates b. It does not run until the thread that computes a fails.

    Note: Thread b remains in place even after thread a exits the stream expression. When a fails, thread b runs, even if a fails after exiting the stream expression. This is where Stream differs from Try. In a try-expression, the waiting thread is destroyed when a thread successfully completes the condition.

    When a stream fails, its exception is that of the last thread to fail.

    For example, 

       Open Stream
     Let x = 1.
   then
     Let x = 2.
   %Stream
   Writeln[$x].
   {false}
    prints 
       1
   2
    and then fails with exception testX. (The first thread prints 1, then fails, causing the second thread to start. That thread prints 2 and then fails. So no threads complete this expression successfully, although the side-effects that they cause before failing are visible.) You can avoid getting the failure at the end by wrapping the entire thing in a try. For example, 
       Try
     Open Stream
       Let x = 1.
     then
       Let x = 2.
     %Stream
     Writeln[$x].
     {false}
   %Try
    also prints 
       1
   2
    but does not fail, since the failure of the second thread is caught (and ignored) by the try. Notice that both the then- and else-parts have been omitted from the try, so both default to ().

    Notice that a stream expression can have a value. An alternative way of writing the previous example is 

       Try
     Let x = Stream 1 then 2 %Stream.
     Writeln[$x].
     {false}
   %Try

    Stream a then b then c %Stream
    This abbreviates Stream a then Stream b then c %Stream %Stream. You can have any number of parts.

    Mix a with b %Mix
    This expression forks the current thread into two. One thread evaluates a, and the other evaluates b. Both threads run in parallel.

    When a mix expression fails, its exception is that of the last thread to fail.

    For example, 

       Try
     Let x = Mix 1 with 2 %Mix.
     Writeln[$x].
     {false}
   %Try
    prints either 
       1
   2
    or 
       2
   1
    or can even print other things, such as 
       21
    (with a blank line after it) since the printing of the numbers and the end-of-line markers can be interleaved in arbitrary ways.

    Mix a with b with c %Mix
    This abbreviates Mix a with Mix b with c %Mix %Mix. You can have any number of parts.

    each(x)
    Function each takes a list as a parameter and produces each member of that list, where each member is produced by a separate thread. The threads are produced using stream, so each runs only when the prior one fails.

    Example: 

          Let x = each [a,b,c].
    is equivalent to 
          Let x = Stream a then b then c %Stream.

    eachMixed(x)
    Function eachMixed is similar to each, but produces threads that run in parallel.

    Example: 

          Let x = eachMixed [a,b,c].
    is equivalent to 
          Let x = Mix a with b with c %Mix.

    From threads to lists

    A stream generally produces several different values, but produces each one on a separate thread. There is a way to get a list of all of the values produced by a stream, where the list is produced by just one thread. If expression e produces a stream of values, then expression [:e:] produces a list of those values. For example, [:Stream 1 then 2 then 3 %Stream:] = [1,2,3].

    If a mixed stream is used, then the values are put into the list in the order in which they are obtained. That order is not defined, but depends on how the implementation runs the threads. For example, [:Mix 1 with 2 with 3 %Mix:] might produce list [3,2,1].

    This feature can be used for list comprehensions, as well as in more general things.

    Atomicity

    Normally, a thread can be interrupted at any point during its execution, to allow other threads to run. You can partially prevent such interrupts by indicating that a section of the program is atomic. The implementation might forbid stopping a thread inside an atomic segment, but it might also simply guarantee that no two sections that are marked atomic can be interleaved with one another. Use expression 
       Atomic
     ... (expression to be done atomically)
   %Atomic
    Do not overuse this. Keep atomic sections short. If you have long atomic sections, you will cripple the thread manager.

    Cutting collections of threads

    Because a thread can fork into several threads, it is possible for a single thread to enter a block of code, and for many threads to come out of that same block. CutHere and Unique expressions provide a mechanism for allowing only one thread to get out of the block. Imagine that expression 
        CutHere
      e
    %CutHere
    is entered by a thread, and that this thread forks into several different thread within this block. If any of those threads performs statement 
        Cut.
    then all other threads that were created inside this CutHere block are destroyed. A thread is destroyed by causing it to fail with exception terminateX.

    A thread can be destroyed even if it has made it out of the CutHere block, as long as some other thread that is still in that block performs a cut.

    A Cut statement applies to the most recently entered CutHere block. It does not need to be enclosed within that block in the static structure of the program, only in the dynamic structure.

    Expression 

        Unique
      body
    %Unique
    is equivalent to 
        CutHere
      body
      Cut.
    %CutHere
    It only allows one thread to exit the CutHere block for each thread that enters the block, regardless of how many are created inside the block.

    Modifying definition by cases

    You can put a CutHere, Unique or Atomic construct around the cases in a definition by cases. For example, the following function computes the larger of two box contents atomically. 
         Define largerContent by
       Atomic
         case largerContent(?a,?b) = @a when @a >= @b
         else largerContent(?a,?b) = b
       %Atomic
     %Define
    The choice mode one mixed is implemented using cut. Definition 
         Define f by one mixed
       case f p1 = v1
       case f p2 = v2
     %Define
    is translated by the compiler into 
         Define f by all mixed
       CutHere
         case f p1 = Cut. v1
         case f p2 = Cut. v2
       %CutHere
     %Define

    Pattern matching and threads

    The following pattern forms are available for creating threads within patterns.
    ?x _orthen_ ?y
    This creates two threads, one matching ?x against the target, the other matching ?y against the target. The threads are created using Stream, so the thread that uses pattern ?y only runs when the thread that uses pattern ?x fails.

    ?x _orwith_ ?y
    This creates two threads, one matching ?x against the target, the other matching ?y against the target. The threads are created using Mix, so the threads run simultaneously.
