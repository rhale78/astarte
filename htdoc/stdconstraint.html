T23. Standard Operations With Constrained Types

Here are some operations that are used with secondary species.
^^
: `a -> `b constraint (`b >= `a)

^^ is a general "upcast" function. It is used to change the extrinsic species of an item without changing its intrinsic species. This function needs to be used when you want to forget information about the species of an item. For example, suppose that you want to build a list of species [ANY], but the things that you want in the list do not already have extrinsic species ANY. Then you can upcast them all. You might write [^^(2), ^^("abc"), ^^(true)] : [ANY].

You can apply operator ^^ to a function to allow the function to apply to a more specific kind of thing than its extrinsic species suggests. For example, suppose that you have a function f:ANY->Natural. You would like to compute f(x), where x has species Real. If you simply write f(x), you get a species mismatch at compile time, since species Real is not the same as species ANY. But if you write (^^f)(x), then all is fine. ^^f generalizes f so that it can take any species as a parameter.

wrap
: `a -> ``b constraint (``b >= `a)

Function wrap converts an item to a secondary species, preserving its intrinsic species. This is a special case of ^^ where the codomain is a secondary species.

rewrap
: ``a ->``b

rewrap(x) changes the extrinsic species of x from one secondary species to another, without changing the intrinsic species of x. Unlike function wrap, it is not restricted to convert the extrinsic species upwards. If conversion is impossible, rewrap fails with exception speciesX. For example, rewrap(<<2:Real>>):INTEGER fails, since it not possible to have an item whose intrinsic species if Real but whose extrinsic species is INTEGER.

safeUnwrap
: ``a -> `*b constraint (``a >= `*b, `*b >>= ``a)

safeUnwrap(x) has the same extrinsic species as the intrinsic species of x. It removes the gift wrapping from x, looking at the intrinsic tag of what it finds inside.

The presence of the light constraint ensures that the unwrap cannot fail due to a species mismatch.

unsafeUnwrap
: ``a -> `*b constraint (``a >= `*b)

This function is identical to safeUnwrap, but it does not have the light constraint, and so the unwrap can fail (with exception speciesX) due to a species mismatch. Use it when you want to test for a given species, or when you think you know more than the compiler about the intrinsic species of a value. For example, if you know that x is a number of some species in genus REAL, you can write 

      Let xval:REAL`a = unsafeUnwrap(x).
This will be allowed by type inference. It will fail with exception speciesX if you are wrong about x.

<<x>>
<<x>> abbreviates wrap(x). When used in a pattern, it uses safeUnwrap to get x. So 
      Match << ?x >> = y.
has the same meaning as 
      Let x = safeUnwrap(y).
Be careful to put a space after << if what follows is a pattern variable. Otherwise, the lexical analyzer will think that <<? is a symbolic identifier.

?<<x>>?
?<<x>>? abbreviates unsafeWrap(x). Function unsafeWrap is the same as wrap when used to compute a value. When used in patterns, it uses unsafeUnwrap to do the unwrapping. So 
      Match ?<< ?x >>? = y.
has the same meaning as 
      Let x = unsafeUnwrap(y).
