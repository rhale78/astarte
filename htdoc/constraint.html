    • The nature of constraints

    You can attach constraints to types. A constraint indicates that one species must be greater than or equal to another in the hierarchy. The meaning of A >= B is that either A = B or A can be obtained from B by replacing species by secondary species in a way that only goes up in the hierarchy. For example,
    • ANY >= REAL (since REAL is a subgenus of ANY).

    • REAL >= Natural (since Natural is a member of REAL).

    • (REAL, ANY) >= (Natural, Natural) (since REAL >= Natural and ANY >= Natural).

    • ANY >= (Natural, ANY) (since species (Natural, ANY) is a member of genus ANY).

    • [ANY] >= [REAL].
    Here are some examples of false inequalities.
    • REAL >= ANY is false, since ANY is not a subgenus of REAL.

    • REAL >= (Natural, Natural) is false, since genus REAL does not include catesian product species.

    • (REAL, ANY) >= (ANY, REAL) is false, since ANY is not a subgenus of REAL.
    You can think of A >= B as saying that A provides less information (or at least no more information) about the type of an item that does B. For example ANY >= REAL because knowing only that an item belongs to species ANY tells you less about the item than knowing that it is a member of species REAL. Generally, if A >= B and you have an item x of species B, then you might reasonbaly expect to be able to use x in a context where an item of species A is expected.

    Constraints involving function types

    Rules for functions are a bit peculiar because of the way function types behave in their domains. For example,

    (REAL->ANY) >= (ANY->REAL).
    This makes sense if you think about it. Suppose that you have a function f:ANY -> REAL. Should you be able to use this in a context where a function of species REAL->ANY is required? The available function f can handle, as its parameter, an item with any intrinsic species at all. But all that is required in the new context is a function that handles items with an intrinsic species that belongs to genus REAL. So function f is more general than we require, and will work.

    When f returns its result, we know that the result has an intrinsic species that belongs to genus REAL. That is fine, since we need something of species ANY, where we do not care what the intrinsic species is.

    Rules for checking constraints

    The following rules indicate those inequalities that are true. Anything that does not follow from these rules is false.
    1. X >= X for all X.

    2. G >= T when species T is a member of genus G.

    3. C >= F when F is a family that is a member of community C.

    4. G >= H when H is a subgenus of G.

    5. C >= D when D is a subcommunity of C.

    6. (A,B) >= (C,D) when A >= C and B >= D.

    7. (A -> B) >= (C -> D) when C >= A and B >= D.

    8. X(A) >= Y(A) when X and Y are opaque communities or families and X >= Y.

    9. X(A) >= Y(B) when X and Y are transparent communities or families and X >= Y and A >= B.

    10. G > X(A) when X is an opaque family or community that is a member of subcommunity of genus G.

    11. G > X(A) when X is a transparent family or community that is a member of subcommunity of genus G, provided G >= A.

    Attaching constraints to types

    To attach a constraint A >= B to a type, just write constraint(A >= B) after the type. For example, the species of function wrap is 
         `a -> ``b constraint(``b >= `a)
    The parentheses around constraint (``b >= `a) are required. The constraint restricts the values that the variables can take on. Only values that satisfy the constraints are allowed. For example, function wrap can have species Natural -> ANY since constraint ANY >= Natural is satisfied, but it cannot have species ANY -> REAL since constraint REAL >= ANY is false.

    When types are printed, each constraint is generally stated only once, the first time one of its variables is encountered.

    If you have multiple constraints, you can use a separate constraint phrase for each one. Alternatively, you can use just one constraint phrase, and separate the constraints by commas. For example, you can write 

          `a -> (`b,`c) constraint (`b >= `a, `c >= `a)
    You can also use <= in constraints if desired. A <= B means the same thing as B >= A.

    Light constraints

    There is a special form of constraint that does not restrict the bindings of variables, but instead indicate how type inference should be done. If you write A >>= B, then the type inference mechanism will make sure that variables in B are sufficiently restrictive that, for every species C that B might bind to, it is possible to bind A to a species D where D >= C.

    For example, if you write `a >>= `b and, through the course of type inference, `a is restricted to domain REAL, then `b will also become restricted to domain REAL. See safeUnwrap for an example.
