D8. Team Declarations

Normally, definitions can only be used after they are made. There are two ways to get around that restriction. One is to use expectations. The other is to use a team.

A team is a collection of definitions that can use one another. For example, the following definition of the factorial function can be made. Notice that factorial uses factorialHelp, even though factorialHelp occurs after factorial in the team. 

    Team
      Define factorial(?n) = factorialHelp(n,1).

      Define factorialHelp by
        case factorialHelp(0,?r) = r
        case factorialHelp(?n+1,?r) = factorialHelp(n,r*(n+1))
      %Define
    %Team
Teams are typically used for mutually recursive definitions.

When a team is compiled, type inference is performed on the team as a whole. If a team is large, then type inference will consume a great deal of memory. It is not a good idea, therefore, just to put all definitions of a large package in a team. Use expectations instead to handle a large collection of mutually recursive functions.

The following kinds of declarations can be put inside a team.

  1. Define declarations.

  2. Let declarations. When inside a team, a let declaration is handled exactly as if it were a define declaration.

  3. Description declarations.

  4. Advise declarations.
Regardless of the order of the declarations, the order of processing is
  1. All advisories are done, in the order written.
  2. All definitions are made.
  3. All descriptions are done.
