    • Variables and Unknowns

    Logic programming uses variables that are bound in the course of computation. Such variables are modeled by unknowns in Astarte. An unknown is an item whose value has not been found. It is not a lazy value, because no function is attached to it that computes the value on demand. An unknown is given a value in a way similar to a box. However, an unknown differs from a box in that once an unknown is given a value, the binding is permanent, and cannot be undone or replaced.

    Unification

    Unification is a form of pattern matching. It attempts to bind unbound unknowns so as to make two items equal. For example, suppose that u1 and u2 are unbound unknowns. To unify [u1,3] with [6,u2], unknown u1 becomes bound to 6 and unknown u2 gets bound to 3.

    Unificaton can bind unknowns to one another, leaving both unbound. When two unknowns u1 and u2 are bound together, any binding of one is also a binding of the other. For example, suppose that you unify [u1] with [u2]. This causes u1 and u2 to be bound to one another. If you then unify [u2] with [3], you find that both u1 and u2 become bound to 3.

    Unification does not always succeed. For example, if you try to unify [u1,3] with [u2,4], unification fails, because 3 and 4 cannot be made the same.

    There are two variants of unification. Imagine unifying u with 1::u, where u is an unbound unknown. That is, we are trying to make u be the same as a list whose head is 1 and whose tail is u.

    1. The first kind of unification, naive unification, succeeds. It binds u to 1::u, yielding a [1,1,1,...], an infinite list of 1's.

    2. The second kind of unification, full unification, fails. In general, full unification will refuse to bind an unknown to an item that contains the same unknown embedded within it.

    To unify, use the following functions. All are provided in logic/unify.ast, and you must import that package to use unification.

    Unify
    : (EQ`a,EQ`a) -> ()

    Unify(x,y) %Unify unifies x and y using naive unification. It fails with exception testX if the unification cannot be done.

    Unify?
    : (EQ`a,EQ`a) -> ()

    Unify?(x,y) unifies x and y using full unification. It fails with exception testX if the unification cannot be done.

    ==!
    : (EQ`a,EQ`a) -> Boolean

    x ==! y unifies x and y using naive unification, and returns true if the unification succeeds. It returns false if the unification cannot be done.

    ==!?
    : (EQ`a,EQ`a) -> Boolean

    x ==!? y unifies x and y using full unification, and returns true if the unification succeeds. It returns false if the unification cannot be done.

    Active pattern matching

    When a pattern variable must match two different items, an equality test is normally used. For example, 
          Match (?x,?x) = (y,z).
    only succeeds if y and z are the same value. This form of pattern matching is passive because it only examines the target (pair (y,z) in this case). You can replace the equality test with unification if desired. Statement 
          Match active(?x,?x) = (y,z).
    causes y and z to be unified with one another. This is an active form of pattern matching because it can bind unknowns in the target in order to make the match succeed. Within an active match, equality tests are replaced by unifications.

    Pattern active(...) uses Unify, and pattern active?(...) uses Unify?.

    Definition by cases

    When you perform a definition by cases, you can request that pattern matching be active. Instead of = in the definitions, use either <- (for naive unification) or <-? (for full unification). For example, here is a function that concatenates two lists in an active form. It binds its third parameter to the concatenation of the first two. 
      Define Append by all
    case Append(nil,    ?a, ?a).     <- ()
    case Append(?h::?t, ?a, ?h::?r). <- Append(t, a, r).
  %Define
    Notice that pattern ?a occurs twice in the first case. This will force a unification to be done when the first case is tried. Similarly, the second case contains ?h twice, and that will lead to a unification. This is an example of logic programming available. You can also use <- or <-? instead of = in definitions that have a single case. They call for unification to be employed in pattern matching. For example, 
      Define Eq(?x,?x). <- ().
    Defines Eq so that it unifies its two parameters together, without an occur check.

    Additional functions

    Additional functions that use unification are available. Each is labeled by the package that must be imported to use it.

    freeze
    : `a -> `a

    There is an important consideration when using nonshared unknowns and threads. When one thread binds a nonshared unknown, which is quite common in logic programming, other threads do not see that binding. This can cause a problem when passing a value from one thread to another, when the value contains bound nonshared unknowns, since the other thread will see those unknowns as unbound.

    freeze(x) is the same as x, but the bindings of unprotected unknowns are frozen. That is, the unknown is removed, and replaced by the value to which it is bound. Use this when you want to pass an object that might contain bound nonshared unknowns from one thread to another.

    hasUnknowns?
    : `a -> Boolean

    hasUnknowns?(x) is true if x has any unbound unknowns in it. Be careful not to apply this to a cyclic structure, since it will not terminate then.

    _in!_
    : (EQ`a, COLLECTION`c(EQ`a)) -> Boolean

    x _in!_ s is true if x is a member of set s. If possible, x will be unified (without an occur check) with each member of s, and all solutions will be returned as a stream. If x has no unknowns, then a lookup using _in_ will be used.

    _inMixed!_
    : (EQ`a, COLLECTION`c(EQ`a)) -> Boolean

    _inMixed!_ is similar to _in!_, but the solution is returned as a mixed stream.

    occurCheck
    : (`a, `b) -> Boolean

    occurCheck(u,x) is true if unknown u occurs inside x. This only goes into the structures of pairs and lists by default to look for u, but you can change that.

    More on unification

    Unification is designed only to look for unknowns inside certain types of items. The default is to look inside lists and ordered pairs. Unification will go to arbitrary depth as long as it only looks inside lists and ordered pairs. For example, if u is an unknown then unifying [(2, [u,5]), (4,3)] and [(2, [6,5]), (4,3)] succeeds, and binds u to 6.

    If you have a different kind of structured species, unification will not look inside the structure for unknowns. Instead, it just does an equality test. If there is an unknown inside such a structured item, the equality test will typically pause its thread, waiting for the unknown to become bound.

    You can cause unification to go into the structure of other kinds of items. To do that, override each of the following functions for your structured species.

    1. Unify
    2. Unify?
    3. hasUnknowns?
    4. freeze
    5. occurCheck
